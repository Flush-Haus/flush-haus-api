# AGENTS.md

## Overview

This project is a Texas Hold'em Poker server developed for a Computer Networks course.

The primary goal of the project is to implement and use a custom communication protocol based on text messages transmitted through WebSockets.

Protocol compliance is more important than optimizations, scalability, or external integrations.

---

# Goals

Implement a server using:

* Bun
* TypeScript
* Elysia
* WebSocket

The server must:

* Manage multiple game sessions simultaneously.
* Execute complete Texas Hold'em matches.
* Strictly follow the project's textual protocol specification.
* Keep all state in memory.
* Not use a database.
* Not use authentication.
* Not use Redis.
* Not use queues.
* Not use microservices.

All state is lost when the process terminates.

# Protocol

## Fundamental Rule

The WebSocket transports only plain text strings.

Example:

```text
session create Heitor
```

```text
game raise 200
```

Never use JSON for communication.

---

# Parser

Every incoming message must pass through:

```text
Raw String

```

Example:

```text

```

becomes:

```ts
{
  domain: "game",
  action: "raise",
  params: ["200"]
}
```

---

# Serialization

Messages sent to clients must be generated by serializers.

Avoid manual string concatenation scattered throughout the codebase.

Examples:

```ts
serializeGameTurn(...)
serializeGameAction(...)
serializeSessionInfo(...)
```

---

# Connection Model

Each WebSocket connection represents a client.

A client may:

* create a session;
* join a session;
* play the game;
* disconnect;
* reconnect.

---

# Reconnection

There is no authentication.

Reconnection must work through the playerId.

Upon reconnection:

* the client provides its playerId;
* the server searches for the player in memory;
* the old connection is replaced;
* the player keeps:

  * chips;
  * seat position;
  * state;
  * game participation.

Reconnection must not restart the match.

# Session States

## Lobby

Initial state.

Allows:

```text
session join
session leave
session players
session ban
session unban
session start
```

---

## Running

A game is currently in progress.

New players:

* may join;
* receive the waiting state.

They participate only in the next round.

---

## Closed

Session has ended.

No actions are allowed.

# Poker Rules

Implement No-Limit Texas Hold'em.

---

## Dealer

Dealer position rotates after each round.

---

## Small Blind

Posted automatically.

---

## Big Blind

Posted automatically.

---

## Turn Order

Follow official Texas Hold'em rules.

---

## Valid Actions

```text
fold
check
call
bet
raise
all_in
```

Validate:

* correct turn;
* minimum amount;
* maximum amount;
* available chips.

---

## Side Pots

Side pots must be implemented correctly.

The implementation must support:

* multiple players;
* multiple all-ins;
* multiple side pots.

Pot distribution must be performed pot-by-pot.

---

## Showdown

Reveal only eligible players.

Send:

```text
game showdown ...
```

before:

```text
game result ...
```

---

## Elimination

A player with:

```text
chips == 0
```

is eliminated.

Send:

```text
game eliminated
```

---

## End of Match

When only one player remains:

```text
game over [winnerPlayerId]
```

# ID Generation

Use human-readable identifiers.

Sessions:

```text
session_000
session_001
session_002
```

Players:

```text
player_000
player_001
player_002
```

Do not use UUIDs.

The goal is to simplify debugging and protocol inspection.

---

# Error Handling

Every error must generate:

```text
error [requestCommand] [errorCode] ...
```

Never silently close a connection.

---

# Timeouts

Implement:

```text
session ping
session pong
```

If a client stops responding:

* mark the player as disconnected;
* preserve all player state.

Do not immediately remove the player from the session.

---

# Acceptance Criteria

The server must successfully execute:

1. Session creation.
2. Multiple player joins.
3. Match start.
4. Complete rounds.
5. Fold.
6. Check.
7. Call.
8. Bet.
9. Raise.
10. All-in.
11. Side pots.
12. Showdown.
13. Player elimination.
14. Reconnection.
15. Match termination.

Everything must operate exclusively through the defined text protocol.

# Development Priority

Implement features in the following order:

1. Basic WebSocket server.
2. Protocol parser.
3. Session management.
4. Player management.
5. Lobby flow.
6. Poker engine.
7. Hand evaluator.
8. Side pots.
9. Reconnection.

# Additional Context

This project is part of a Computer Networks course.

The custom protocol is one of the primary deliverables and must be treated as a first-class component of the system.

Implementation decisions must never alter, simplify, or bypass the protocol specification for development convenience.

The protocol itself is more important than framework-specific optimizations.
